# Phase 2: hubflora-media — DB Ownership & Public-Facing Auth

## Overview

Transform hubflora-media from a stateless file processor into a fully autonomous media
microservice that owns persistence, validates user identity via Better Auth JWTs, and
serves browser clients directly.

## Goals

1. **Performance** — Eliminate the upload round-trip where traveler-aggregator persists
   media metadata after every Go service call. The Go service handles it end-to-end.
2. **Independence** — Any service (or browser) can use the media service without needing
   its own media tables.

## 1. Authentication

### 1.1 Dual Auth Middleware

The Go service accepts two authentication methods. JWT takes priority.

**Path 1 — JWT (browser clients):**
- Extract `Authorization: Bearer <token>` header
- Validate signature via JWKS fetched from `{JWKS_URL}/api/auth/jwks`
- Validate `iss` and `aud` claims match `BETTER_AUTH_URL`
- Validate `exp` (token not expired)
- Extract claims: `sub` (userId), `orgId` (organizationId), `orgSlug`
- Inject into request context

**Path 2 — API key (server-to-server):**
- Extract `X-Media-API-Key` header, validate against `MEDIA_SERVICE_API_KEY` env var
- Extract `X-Media-User-Id`, `X-Media-Org-Id`, `X-Media-Org-Slug` from headers
- Inject into request context

**Path 3 — Neither:** Return `401 Unauthorized`

### 1.2 JWKS Caching

- Cache JWKS in-memory with 1-hour TTL
- On unknown `kid` in a JWT header, re-fetch JWKS (handles key rotation)
- Use Go `jose` library (`go-jose/go-jose/v4`) for JWT parsing/validation
- Fallback: if JWKS endpoint is unreachable, use cached keys until TTL expires

### 1.3 Request Context

Both auth paths produce the same context struct:

```go
type AuthContext struct {
    UserID         string
    OrganizationID string
    OrgSlug        string
    AuthMethod     string // "jwt" or "apikey"
}
```

All handlers extract this from `r.Context()` — no handler ever reads auth headers directly.

### 1.4 New Config Fields

```
BETTER_AUTH_URL=https://auth.hubflora.com    # JWKS fetched from {this}/api/auth/jwks
DATABASE_URL=postgresql://user:pass@host:5432/hubflora
```

`MEDIA_SERVICE_API_KEY` remains for server-to-server auth.

## 2. Database Integration

### 2.1 Shared Database, Schema Owned by Traveler-Aggregator

The Go service connects to the **same PostgreSQL database** as traveler-aggregator.
Schema changes are made via Drizzle ORM migrations in traveler-aggregator.
The Go service is a consumer of the schema, not a manager.

### 2.2 Driver

- `github.com/jackc/pgx/v5/pgxpool` — connection pool
- Raw SQL queries (no ORM) — the media tables are simple (2 tables, straightforward CRUD)
- Prepared statements for performance

## 3. Handler Changes

### 3.1 Upload (POST /api/v1/media/upload)

**Before (Phase 1):** Upload file to S3 → return metadata DTO → caller persists to DB.

**After (Phase 2):** Upload file to S3 → persist to DB → return complete record.

Flow:
1. Parse multipart form (file + optional alt/caption/description)
2. Validate MIME type
3. Extract orgSlug from auth context (JWT claims or API key headers)
4. Generate object key: `{orgSlug}/{timestamped-folder}/original.{ext}`
5. Upload to S3
6. Extract image metadata (dimensions, format)
7. Generate UUID for media_files record
8. **INSERT into media_files** (new)
9. If `generateVariants && isImage`:
   - If `async`: enqueue asynq task with `mediaFileID` → **INSERT variant records when task completes**
   - If sync: process variants → **INSERT variant records** → set `thumbnail_url`
10. **UPDATE media_files** with thumbnail_url if variants generated (new)
11. Return complete `UploadResponse` with DB record ID

### 3.2 Delete (DELETE /api/v1/media/)

**Before:** Delete from S3 only.

**After:** Delete from DB + S3.

Flow:
1. Extract `id` or `objectKey` from request
2. Verify record exists and belongs to auth context's org (**org-scoped query**)
3. **DELETE from media_variants** WHERE media_file_id = id
4. **DELETE from media_files** WHERE id = id
5. Delete from S3 (folder prefix)
6. Return success

### 3.3 Crop (POST /api/v1/media/crop)

**Before:** Crop file in S3, return metadata.

**After:** Crop file in S3, **UPDATE media_files** record (new dimensions, file size), optionally regenerate variants and **UPDATE variant records**.

### 3.4 Variant Regeneration (POST /api/v1/media/variants)

**Before:** Generate variants in S3, return variant info.

**After:** Generate variants in S3, **INSERT/UPDATE media_variants** records.

### 3.5 Async Worker Changes

The asynq worker that processes variant generation tasks now:
1. Downloads original from S3
2. Generates all variant images
3. Uploads variants to S3
4. **Batch INSERT into media_variants** table
5. **UPDATE media_files** set thumbnail_url = thumbnail variant URL

## 4. New API Endpoints

These endpoints are new in Phase 2 — they query the database.

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/v1/media/:id` | Get media file by ID (org-scoped) |
| `GET` | `/api/v1/media` | List media files for org (paginated, searchable) |
| `POST` | `/api/v1/media/batch` | Get media files by IDs (org-scoped) |
| `PATCH` | `/api/v1/media/:id` | Update metadata (alt, caption, description, isPrivate) |
| `GET` | `/api/v1/media/:id/variants` | Get variants for a media file |

### 4.1 GET /api/v1/media/:id

Returns a single media file record with variants. 404 if not found or wrong org.

### 4.2 GET /api/v1/media

Query parameters:
- `limit` (default 50, max 100)
- `offset` (default 0)
- `search` (searches filename, alt, caption)
- `type` (image, video, audio, document)
- `sort` (created_at, file_size, filename)
- `order` (asc, desc)

Returns `{ items: MediaFile[], total: number }`.

### 4.3 POST /api/v1/media/batch

Body: `{ ids: ["id1", "id2", ...] }` (max 100)

Returns `{ items: MediaFile[] }` — only items belonging to the auth context's org.

### 4.4 PATCH /api/v1/media/:id

Body: `{ alt?, caption?, description?, isPrivate? }`

Updates only the provided fields. Sets `updated_at = now()`.

### 4.5 GET /api/v1/media/:id/variants

Returns `{ variants: MediaVariant[] }` for the given media file.

## 6. CORS Changes

The Go service becomes public-facing. CORS must allow browser requests from org subdomains.

```go
cors.Handler(cors.Options{
    AllowedOrigins:   cfg.AllowedOrigins, // ["https://*.hubflora.com", "http://*.lvh.me:3000"]
    AllowedMethods:   []string{"GET", "POST", "PATCH", "DELETE", "OPTIONS"},
    AllowedHeaders:   []string{"Authorization", "Content-Type", "X-Media-API-Key", ...},
    AllowCredentials: false, // Bearer tokens, not cookies
    MaxAge:           300,
})
```

Note: `AllowCredentials: false` because we use `Authorization: Bearer` header, not cookies.
The SDK explicitly sends the token — no cookie-based auth needed.

### 7.2 Auth Header Logic

```typescript
private async authHeaders(): Promise<Record<string, string>> {
  if (this.tokenProvider) {
    const token = await this.tokenProvider();
    return { Authorization: `Bearer ${token}` };
  }
  if (this.apiKey) {
    return { "X-Media-API-Key": this.apiKey };
  }
  throw new Error("HubfloraMedia: either apiKey or tokenProvider is required");
}
```

### 7.3 Upload Options Change

## 8. React Provider Changes (@hubflora/media-client/react)

### 8.1 Session-Synced Provider

The React provider ensures the Better Auth session's `activeOrganizationId` is
synced before any media operation occurs. This closes the gap where a user switches
org subdomains but the JWT still contains the old org's ID.

```typescript
interface HubfloraMediaProviderProps {
  children: React.ReactNode;
  /** Media service base URL */
  baseUrl: string;
  /** Function that syncs the org session and returns a fresh JWT access token */
  getToken: () => Promise<string>;
  /** Current organization ID — triggers re-sync when it changes */
  organizationId: string;
  /** Fallback UI while syncing (default: null) */
  fallback?: React.ReactNode;
}
```

The provider:
1. On mount and when `organizationId` changes:
   - Calls `getToken()` which should sync the org session first, then return the JWT
2. Creates a `HubfloraMedia` client with a `tokenProvider` that returns the cached token
3. Refreshes the token before expiry (e.g., every 12 minutes for a 15-minute token)
4. Renders children only after the first token is obtained

### 8.2 Integration in traveler-aggregator

```typescript
// In the app layout or media-enabled pages:
<HubfloraMediaProvider
  baseUrl={process.env.NEXT_PUBLIC_MEDIA_SERVICE_URL}
  organizationId={org.id}
  getToken={async () => {
    // This calls Better Auth's token endpoint
    // which returns a signed JWT with orgId + orgSlug claims
    const res = await authClient.$fetch("/token");
    return res.data.token;
  }}
>
  {children}
</HubfloraMediaProvider>
```

## 11. Migration Strategy

### Step 1: Add DB + auth infrastructure
- Add pgx dependency, repository layer, config changes
- Add JWKS-based JWT validation middleware
- Deploy with new env vars (`DATABASE_URL`, `BETTER_AUTH_URL`)
- Existing API key auth continues to work (backward compatible)

### Step 2: Make handlers persist to DB
- Upload handler inserts into media_files + media_variants
- Delete handler removes from DB + S3
- Crop/variant handlers update DB records
- Asynq worker persists variant records

### Step 3: Add new query endpoints
- GET /api/v1/media/:id
- GET /api/v1/media (list)
- POST /api/v1/media/batch
- PATCH /api/v1/media/:id

### Step 4: Update SDK
- Add dual auth support (tokenProvider + apiKey)
- Add new methods (get, list, batchGet, update)
- Update React provider with session sync
- Publish new SDK version

### Step 5: Verify
- All existing API key flows still work
- New JWT flows work from browser
- DB records match S3 state
- Traveler-aggregator can still read media tables directly for joins
